# Audit C2 — Form-4 / EDGAR insider ingestion

- **Lane:** C2 (Form-4 / EDGAR insider ingest path)
- **Date:** 2026-06-19
- **Auditor:** read-only (no source/test/config modified)
- **Scope:** `arbiter/ingest/edgar/{client,parser,normalize}.py`, `arbiter/ingest/runner.py` (form4 path), `arbiter/signals/detection.py` (cluster/single detectors), `arbiter/ingest/writer.py` (dedup). Code + tests reviewed; no live EDGAR hit.
- **Test status:** `tests/ingest/test_edgar.py` — 80 passed in 0.19s.

## VERDICT

**NOT production-ready when enabled (P0 present).** The unit-tested pieces (parser, normalize, dedup, 10b5-1 exclusion, rate-limit etiquette, retry/backoff, the `ValueError`-on-empty-UA guard) are correct and well covered. But the **live HTTP discovery path (`search_form4_filings`) has never been exercised against real EDGAR and is built on incorrect assumptions** — it will return zero (or wrong) filings the moment `EDGAR_USER_AGENT` is set. Net effect: enabling Form-4 silently ingests nothing useful while reporting "success." Two systemic gaps compound this: EDGAR's 403-on-bad-UA is not retried/surfaced, and the 10b5-1 detector misses the post-2023 structured checkbox.

## FINDINGS

### P0

**[P0] — `search_form4_filings` queries EDGAR by company NAME with a ticker, and parses an atom schema EDGAR does not emit — discovery returns nothing live — `arbiter/ingest/edgar/client.py:120-137, 205-234` — ** The URL is `/cgi-bin/browse-edgar?action=getcompany&company={ticker}&type=4&output=atom`. `company=` is a company-NAME substring match, not a ticker; passing `AAPL` will not resolve to Apple Inc. The correct param is `CIK={ticker}` (browse-edgar accepts a ticker in the `CIK` field) or the submissions JSON API. Worse, `_parse_edgar_atom` looks for `{http://www.sec.gov/Archives/edgar}accession-number` and `cik` child elements — the browse-edgar getcompany atom feed does NOT contain those elements (accession lives inside `<id>`/`<content>` differently, and there is no per-entry `cik` element). Every entry will yield `accession=""`/`cik=""`, which the runner then drops at `runner.py:253` (`if not accession or not cik`). **Net: form4 ingests 0 filings on first live run, while `IngestSummary` reports n_fetched=0/n_written=0 with no error.** This entire path is mocked in tests (`test_edgar.py` only mocks `get_form4_xml`/`_get`; there is NO test that feeds a real browse-edgar atom sample through `_parse_edgar_atom`), so the bug is invisible to the suite. — **Action:** Replace discovery with the documented EDGAR submissions API (`https://data.sec.gov/submissions/CIK##########.json`) or full-text-search API (`https://efts.sec.gov/LATEST/search-index?q=...&forms=4`), resolve ticker→CIK via the official `company_tickers.json` map, and add a test that parses a *real captured* EDGAR response (not a hand-rolled fixture matching the parser's own assumptions).

### P1

**[P1] — Form-4 ingest never runs in the engine cycle; only via manual `arbiter ingest` CLI — `arbiter/cli.py:105-126` vs `arbiter/engine.py:938` — ** `run_ingest` is invoked ONLY from the `ingest` CLI command. The engine's `run`/cycle path (`engine.py:146,176,938`) calls `detect_signals` directly against whatever is already in the DB and never calls `run_ingest`. So even once `EDGAR_USER_AGENT` is set, the live trading loop does not pull fresh Form-4 data unless an operator separately runs `arbiter ingest` on a schedule. This is the "skipped every run" gap: today it is skipped because the UA is empty (`runner.py:197`); tomorrow it is skipped because nothing in the hot loop calls the ingest at all. — **Action:** Decide and document the ingest cadence: either wire `run_ingest` into the engine cycle (guarded by a min-interval to respect EDGAR rate limits) or ship a cron/`deploy` job that runs `arbiter ingest` and assert that contract in a test.

**[P1] — EDGAR's bad-User-Agent / blocking response is 403, which is NOT retried and surfaces as a hard failure — `arbiter/ingest/edgar/client.py:147-162` — ** Retry/backoff only triggers on `{429, 503}`. When SEC throttles or rejects a malformed UA it commonly returns **403 Forbidden** (and sometimes 403 on sustained abuse), which here hits `resp.raise_for_status()` → `HTTPStatusError` → caught as `httpx.HTTPError`, slept, retried 3× with no benefit, then `EdgarError`. The first live run with a slightly-off UA string (EDGAR requires `Sample Company Name AdminContact@sample.com` format) will fail every ticker. The module docstring claims "10 requests per second" tolerance but never validates UA shape. — **Action:** Treat 403 explicitly (do NOT blind-retry; log the UA-format requirement and abort the source with a clear note), and optionally validate `edgar_user_agent` contains an `@`/contact at `EdgarClient.__init__`.

**[P1] — 10b5-1 detection misses the structured `<aff10b5One>` checkbox used by Form 4 since 2023 — `arbiter/ingest/edgar/parser.py:100-120` — ** `_detect_10b5_1` keys off `<planName>` text and footnotes containing "10b5". Since the Dec-2022/2023 SEC amendments, Form 4 carries a structured affirmative flag `<aff10b5One>1</aff10b5One>` (and a related `<noSecuritiesOwned>`-style block); many filers populate the checkbox WITHOUT a `<planName>` element or a "10b5" footnote. Those plan trades will pass the filter and be ingested as discretionary buys/sells, polluting the cluster/single-insider signals (the whole point of excluding 10b5-1 is to drop pre-scheduled, non-informative trades). — **Action:** Add `<aff10b5One>` (truthy value `1`/`true`) to the detection in `_detect_10b5_1`; add a fixture exercising the checkbox-only case.

### P2

**[P2] — `_extract_xml_filename` grabs the FIRST `.xml` link on the index page, which may not be the primary Form-4 document — `arbiter/ingest/edgar/client.py:186-202` — ** The regex returns the first `*.xml` href. EDGAR filing index pages can list multiple XML artifacts (e.g. the rendering/`R*.xml`, `Financial_Report.xml`, or `*-index-headers` variants); grabbing the wrong one yields XML the parser silently produces 0 rows from (no `<nonDerivativeTransaction>`). Untested against a real multi-document index. — **Action:** Prefer the entry whose `type` column is `4`/`4/A` or whose filename matches the ownership-doc pattern; fall back to the submissions-API `primaryDocument` field rather than HTML scraping.

**[P2] — `filing_ts` resolves to `periodOfReport` at midnight UTC, but signals/dedup treat it as the information-availability time — lag/look-ahead risk — `arbiter/ingest/edgar/parser.py:78-97` — ** `_parse_filing_ts` prefers `periodOfReport` (the TRANSACTION date) over `filingDate`, anchored to 00:00 UTC. INTERFACES.md §3 (line 106) states "Form 4 = filing timestamp." Using the transaction date as `filing_ts` understates the real lag (insiders have ~2 business days to file): a trade dated Mar-15 but filed Mar-17 becomes information "as of" Mar-15 in the DB, while it was not public until Mar-17. Because `detect_signals` filters `filing_ts <= as_of` (`detection.py:175`) and skips look-ahead on `latest_ts > as_of` (`detection.py:294`), this can let a signal fire ~2 days before the data was actually knowable — a backtest look-ahead bias and a contract mismatch with INTERFACES.md. Midnight-UTC anchoring also means a same-day cluster window is correct but the 00:00 anchor can shift a trade across a day boundary vs. ET. — **Action:** Set `filing_ts` from `filingDate` (the public-availability date) per the INTERFACES contract, and keep `periodOfReport` in `raw_json` for analytics. If transaction-date semantics are intended, amend INTERFACES.md and document the look-ahead implications.

**[P2] — `transaction_code` is trusted blindly; an "A"/"D" acquired-disposed mismatch is not cross-checked, and code "P" is always treated as a buy regardless of `acquiredDisposedCode` — `arbiter/ingest/edgar/parser.py:171-217`, `normalize.py:48-94`, `detection.py:187` — ** The parser captures `acq_disp` but never validates it against `transaction_code`; normalize keeps only `P`/`S` and detection treats `txn_type='P'` as a buy. A malformed/edge filing with `transactionCode=P` but `acquiredDisposedCode=D` (or vice-versa) would be miscategorized. Real Form 4 data is generally consistent here, so impact is low, but there is no guard and no test for the inconsistent case. — **Action:** Cross-check `acq_disp` against the code (P/A=acquire, S/D=dispose) and drop or log rows that disagree.

**[P2] — Reporting-owner handling takes only the FIRST `<reportingOwner>`; multi-owner filings lose people, weakening cluster detection — `arbiter/ingest/edgar/parser.py:157-163` — ** A single Form 4 can list multiple reporting owners (joint filers). The parser uses `root.find(".//reportingOwner)` (first only) and applies that `person_id`/`person_name` to ALL transactions in the doc. Since `cluster_min_people` counts DISTINCT `person_id`, collapsing co-filers can undercount a genuine cluster (or, conversely, attribute one owner's transactions to another). — **Action:** Map each transaction to its actual owner where the schema permits, or at minimum document the single-owner assumption.

### P3

**[P3] — `_rate_limit` is per-`EdgarClient`-instance, not per-process/per-IP; concurrent clients can exceed 10 req/s — `arbiter/ingest/edgar/client.py:164-169` — ** `_last_request_ts` is instance state. The runner creates one client per `run_ingest` call (fine today, single-threaded), but the engine's threaded cycle (`engine.py:146,176` run per-thread) or any future parallel ingest would each get their own limiter and collectively breach EDGAR's 10/s IP cap → 403/429 ban. — **Action:** If ingest ever goes concurrent, move the limiter to a process-global (module-level lock + timestamp) or a shared token bucket.

**[P3] — `_rate_limit` mixes `time.monotonic()` with the injected `sleep_fn` but reads the real clock — minor test-determinism / drift seam — `arbiter/ingest/edgar/client.py:164-169` — ** `sleep_fn` is injectable for tests, but `time.monotonic()` is called directly, so the limiter's elapsed calc is not fully controllable in tests and the first call always computes a large `elapsed` (init `_last_request_ts=0.0`) — harmless but means the very first request never sleeps even mid-burst across clients. — **Action:** Inject a `monotonic_fn` alongside `sleep_fn` for deterministic limiter tests.

**[P3] — Discovery `count=20` per ticker with no date filter (`dateb=` empty) — re-runs re-fetch the same 20 newest each time — `arbiter/ingest/edgar/client.py:120-134`, `runner.py:243` — ** Dedup at the writer level makes this correct but wasteful: every run pulls and re-parses the latest 20 filings per ticker (10 tickers = 200 detail fetches), most discarded as duplicates. The `lookback_days` param is documented as "less relevant" for EDGAR and is unused on the form4 path. — **Action:** Once the submissions API is adopted, filter by date/accession high-water-mark to fetch only new filings.

## OPPORTUNITIES TO ADD

1. **Ticker→CIK resolution layer.** There is no mapping from watchlist tickers to CIKs; EDGAR is CIK-native. Adopt the official `company_tickers.json` (cached) so discovery is correct and stable. This is a prerequisite for fixing the P0.
2. **A real-response integration fixture.** Capture one genuine EDGAR submissions/search response and one real Form-4 XML (namespaced and un-namespaced variants exist) into a fixture and assert the FULL discovery→parse→normalize→write path. The current suite tests each unit against fixtures shaped to its own assumptions, which is exactly why the P0 discovery bug is invisible.
3. **Derivative transactions are intentionally skipped (`parser.py:142-145`).** Option exercises/grants (codes M/A) can be highly informative (e.g. cashless exercise + immediate open-market sale). Consider a future lane to surface large derivative activity rather than dropping it entirely.
4. **Amendment (4/A) supersede is wired in the writer but the form4 path never sets `amendment_accession`** (`writer.py:222`, `runner.py:281` calls `write_filing` without it). Verify amendments actually supersede the corrected original end-to-end, or document that 4/A is stored as a sibling row.
5. **Surface "ingested 0 filings" as a health signal.** A successful run that writes nothing across all tickers (the current P0 symptom) should raise a WARNING/alert, not look like success — add a guard in `run_ingest` that flags `n_written==0 && n_fetched==0` for an enabled source.
6. **Validate UA format at construction** (contains a contact email) so a misconfigured `EDGAR_USER_AGENT` fails fast with a clear message instead of 403-looping live.
