# F1 — Daemon Loop Resilience Audit

**Auditor lane:** F1 (daemon loop resilience)
**Date:** 2026-06-19
**Scope:** `arbiter/runtime/daemon.py`, `deploy/com.arbiter.daemon.plist`, `scripts/schedule.sh`.
Out of scope (other lanes): SQLite concurrency (F2), crash recovery (F3), calendar correctness (F4).

**Verdict:** PASS WITH CONCERNS. The loop is fundamentally sound — one bad iteration cannot kill it (broad `except` + bounded exponential backoff that resets on a clean iteration), SIGTERM/SIGINT exit gracefully via the injected `stop_event`, the flock prevents concurrent instances and is released by the OS on crash, the missed-slot rule fires once per `(date, slot)`, and the closed-market sleep is capped at 15 min. The 8 daemon tests pass. No P0. The notable gaps are **observability during failure** (heartbeat is NOT rewritten on a failing iteration, so it goes stale during exactly the window an operator is watching — P1) and an **operator-control caveat** (KeepAlive=true means a plain `kill` is auto-relaunched; only `launchctl bootout` truly stops it — P2, partially documented). A latent **busy-spin** exists if the calendar ever reports closed with a missing/past `next_open` (P2).

---

## FINDINGS

### P1 — Heartbeat is not written on a failing iteration; liveness goes stale during the failure window — `arbiter/runtime/daemon.py:191,201,204-210` — The `except` branch (the wedged-after-error state's own handler) increments backoff and sleeps but never calls `_heartbeat(...)`. `_heartbeat` is only reached on the two success paths (open `:191`, closed `:201`). So if the engine raises every iteration (a persistent broker/DB fault), the heartbeat file freezes at its last-good timestamp and `backoff_s` is never published, even though the loop is alive and backing off. An operator (or a watchdog) polling the heartbeat to detect "daemon stuck" sees a stale `now` indistinguishable from a hung/dead process — the one signal you most want during a fault is suppressed. — **Recommended action:** write a heartbeat in the `except` block too, with `iteration_kind="error"`, the exception string, and the current `backoff_s`, so a degraded-but-alive daemon is observably distinct from a dead one. Guard the heartbeat write itself (it already swallows its own exceptions, so it is safe to call from the handler).

### P2 — KeepAlive=true makes the daemon un-killable by ordinary means; operator can't just `kill` it — `deploy/com.arbiter.daemon.plist:44-45` — `KeepAlive=true` + `ThrottleInterval=30` is the correct long-running pattern, but it has a sharp operational edge: a `kill <pid>` (even `kill -9`) is treated as a crash and launchd relaunches within 30 s. SIGTERM handling in the daemon is graceful (`:294-306` sets `stop_event`), but a graceful SIGTERM delivered by `kill` still results in relaunch. The ONLY way to actually stop it is `launchctl bootout gui/$UID .../com.arbiter.daemon.plist` (which `scripts/schedule.sh uninstall-daemon` does, `:139-152`). This is correct behavior but is a foot-gun in an incident: an operator trying to halt trading with `kill` will watch it come back. — **Recommended action:** the plist comment (`:13-16`) mentions `launchctl bootout` but should state explicitly that `kill`/`kill -9` will be auto-relaunched and that `uninstall-daemon` (or `launchctl bootout`) is the only stop. Better: pair with the kill-switch — document that the broker-side kill switch (`engine._safety_gate`, consulted every iteration) is the in-incident "stop trading without stopping the process" lever, since the loop keeps polling but trades nothing while halted.

### P2 — Potential busy-spin when closed with a missing or past `next_open` — `arbiter/runtime/daemon.py:266-274` — `_until_next_open_capped` returns `1.0` second when `next_open <= now` (`:272-273`). With the live AlpacaCalendar `next_open` is populated, but the offline fallback (`market_calendar.py:191,199`, used on every clock fetch/parse failure — out of lane to fix, but the daemon must tolerate it) can return a session whose `next_open` is stale or in the past. In that state the loop sleeps 1 s repeatedly: a 1 Hz busy-loop that re-hits the (possibly failing) calendar and reconcile each tick — CPU/log churn and, if the calendar call is the failing one, it interacts with the backoff path. The `1.0` floor was chosen to avoid a zero/negative sleep, but it is not bounded upward toward sanity. — **Recommended action:** raise the floor for the "next_open in the past/None" case to a small fixed re-poll interval (e.g. 30-60 s) rather than 1 s, so a degraded calendar cannot drive a hot loop. A past `next_open` means "I don't know when we open" → poll politely, not aggressively.

### P3 — Missed-slot key recomputes the ET date after `_run_full_cycle`, risking a midnight-boundary mismatch — `arbiter/runtime/daemon.py:184-186` — The due-slot list is computed from `_due_full_slots(now, ...)` using `_et_now(now).date()` (`:127`), then after running the cycle the fired key is recorded with a *second* `_et_now(now).date()` call (`:186`). `now` is the same value, so today this is consistent. But it is a fragile duplication: if `_run_full_cycle` ever advanced the clock, or `now` were re-read, the recorded date could differ from the due-check date, causing a slot to re-fire. — **Recommended action:** compute `today = _et_now(now).date()` once before the loop over due slots and reuse it for both the due check and the `fired_full_slots.add((today, slot))`, eliminating the second derivation.

### P3 — Lock fd in `main()` is closed but the pidfile is not removed; on hard crash the file lingers — `arbiter/runtime/daemon.py:317-332` — The flock itself is correctly released by the OS when the process dies (advisory flock survival is not the issue — a crashed daemon's lock IS freed, so a fresh instance can re-acquire; this is the desired property). However, the `finally` only `close()`s the fd (`:330`) and never `os.unlink(pidfile)`. The stale pidfile remains with the old PID written in it. This is cosmetic for locking (the next daemon re-`flock`s the same path fine), but any external tooling that reads the pidfile to find "the running PID" will read a dead PID after a crash. — **Recommended action:** in the `finally`, after closing, best-effort `os.unlink(pidfile)` (swallow errors), or write the heartbeat as the authoritative liveness source and document the pidfile as lock-only / not a source of truth.

### P3 — Post-close sweep can be skipped if the open→closed transition iteration raises — `arbiter/runtime/daemon.py:198-203` — The C6 post-close sweep fires only when `state.last_session_open` is True at the moment a closed session is observed (`:198`). It then sets `last_session_open = False` (`:200`). If the iteration that *first* observes the close raises before reaching `:198` (e.g. `calendar.session()` itself throws), the loop hits the `except` (`:204`), `last_session_open` stays True, and on the next iteration the sweep still fires — good. BUT if the throw happens *after* `_run_post_close_sweep` partially runs and before `:200`, the next closed iteration re-runs the sweep. The sweep's sub-steps (`:243-263`) are individually try-wrapped and the outcome sweep is idempotent, so a double-run is tolerated, not corrupting — but it is undocumented reliance on downstream idempotency (F3 territory). — **Recommended action:** set `state.last_session_open = False` *before* invoking `_run_post_close_sweep`, so the transition is recorded as consumed even if the sweep raises; the daily 18:30 one-shot is the documented backstop for a sweep that didn't complete.

## OPPORTUNITIES TO ADD

- **Heartbeat staleness watchdog.** Nothing today alerts when the heartbeat stops advancing. A tiny launchd `StartInterval` job (or the existing daily one-shot) could read `data/arbiter-daemon.heartbeat`, compare `now` against wall-clock, and fire the alert webhook if it is older than, say, `3 × fast_interval_s`. This turns the heartbeat from a manual `daemon-status` artifact into an actual liveness alarm. (Complements the P1 fix.)
- **Max-consecutive-failures circuit.** Backoff is bounded at 600 s but unbounded in *count* — the loop will retry a permanently-broken engine forever at 10-min intervals, hammering the broker/EDGAR on each attempt. Consider a consecutive-failure counter that, past a threshold, fires a critical alert and either widens the cap or pauses the engine (loop stays alive, polling, but stops attempting trades) so a wedged dependency raises a human, not just log lines.
- **Heartbeat schema/version field.** Add a `"schema": 1` and `"pid": os.getpid()` to the `_hb` payload (`:282-291`) so external consumers can validate and correlate the file to the running process (ties into the P3 stale-pidfile finding).
- **Explicit "stop trading vs stop process" runbook.** Given the KeepAlive caveat (P2), a one-paragraph runbook in `deploy/` mapping incidents → levers (kill switch = stop trades, keep polling; `uninstall-daemon` = stop process; pause latch = persistent halt) would prevent the `kill`-then-confusion failure mode in a live incident.
- **Jitter on backoff.** If multiple daemons or the one-shot ever contend, the fixed `30 → 60 → 120 …` doubling synchronizes retries against the same failing dependency. A small random jitter spreads retry load. Low priority given single-instance flock, but cheap insurance.
