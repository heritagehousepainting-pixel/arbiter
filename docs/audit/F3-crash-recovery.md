# F3 — Crash Recovery & Durable-State Consistency (READ-ONLY audit)

Date: 2026-06-19
Auditor lane: F3 (crash/relaunch survival; in-memory vs durable divergence)
Scope: durable `paused` flag (migration 024), `position_store` seed/snapshot,
reconcile-on-start, SimExecutor in-memory vs durable `sim_positions`, daemon
KeepAlive relaunch. (Loop = F1, SQLite mechanics = F2 — out of scope.)

Verdict: **NOT CLEAN.** One P0 (sim state silently rolls back / resurrects a
closed position after a fast-iteration exit + crash), one P1 double-submit
window on alpaca_paper, plus P2/P3 fidelity gaps. The durable `paused` flag is
correct. The headline risk is that the daemon's frequent `run_fast_iteration`
path mutates the in-memory SimExecutor but never snapshots it, so a crash before
the next FULL `run_cycle` restores stale positions/cash from `seed_executor`.

---

## Findings

### [P0] — Fast-iteration sim exits are never snapshotted → closed position resurrects + cash/PL roll back on relaunch — arbiter/engine.py:644-685 (no snapshot) vs arbiter/engine.py:1173-1179 (only snapshot site) — why — `run_fast_iteration` runs the exit monitor (`_run_exit_monitor`, line 682), which for SimExecutor synchronously fills stop-loss / horizon SELLs: the in-memory SimExecutor mutates (position deleted, cash up, `realized_pl` up) AND the durable `ideas`/`outcomes`/`orders` tables are updated via `close_idea_on_sell_fill`. But `snapshot_executor` is called ONLY at the end of `run_cycle` (line 1175), explicitly NOT in `run_fast_iteration` (its own docstring line 653 says "Does NOT ... snapshot"). The daemon (daemon.py:189) calls `run_fast_iteration` every `fast_interval_s` (~180s) whenever `OfflineMarketCalendar.session().is_open` is True — which is True for SimExecutor during regular ET hours (market_calendar.py:88-98 is backend-agnostic), and FULL `run_cycle` runs only at the 2 configured ET slots. So there is a multi-hour window where sim exits have hit the durable ledger but `sim_positions`/`sim_account` still hold the pre-exit snapshot. The daemon's `KeepAlive=true` relaunch (daemon.py:9, 314) rebuilds the engine, and `build_engine` → `seed_executor` (engine.py:1336-1338) restores the STALE snapshot: the just-closed position reappears in-memory, cash and `realized_pl` revert, while `ideas` shows the idea CLOSED and `outcomes` has the realized outcome. Result: a position that was sold is now held again in the sim broker (it can be re-exited → double outcome), and `arbiter status` realized_pl regresses. This is the exact in-memory↔durable divergence this lane exists to catch. — recommended action — snapshot the SimExecutor at the end of `run_fast_iteration` too (mirror the `isinstance(self.executor, SimExecutor)` block from run_cycle), OR snapshot immediately after any sim SELL fill inside `_close_out_filled_sell` / the exit-monitor sim path so the durable snapshot can never lag a durable ledger mutation. Snapshotting inside the same logical step as the ledger mutation is the only fully crash-safe ordering.

### [P1] — Double-submit window: broker `place()` precedes the local-ledger INSERT, and the broker idempotency check is position-presence only — arbiter/execution/submit.py:301 (place) vs :339 (insert) + arbiter/execution/idempotency.py:73-98 (`_check_broker`) — why — In `submit_order` the broker order is placed at line 301 and only afterward inserted into `orders` (line 339). For alpaca_paper a crash (or the broker-fatal re-raise at engine.py:1088, or any exception) between `place()` and the INSERT leaves a real broker order with NO local ledger row. On relaunch there is no reconcile-on-start (see P2), so the next FULL cycle re-evaluates the same idea; `ensure_not_duplicate` checks (a) local ledger — empty, the row was never written — and (b) `_check_broker`, which only returns True if the ticker already shows an open *position* (idempotency.py:87). An ACCEPTED-but-unfilled (`pending`) order produces no position, so both checks pass and the engine submits a SECOND order for the same idea. (SimExecutor is immune — `place` fills synchronously and never crashes mid-call; this is an alpaca_paper hazard.) — recommended action — persist the order row as `pending` BEFORE calling `place()` (write-ahead intent), then update status from the broker report; reconcile-on-start (P2) then promotes/cancels it. Alternatively pass the order's deterministic `dedup_hash` as the Alpaca `client_order_id` so the broker itself rejects the duplicate. Position-presence is structurally insufficient for in-flight unfilled orders.

### [P2] — No reconcile-on-start: divergences are only detected mid-cycle, so a crashed-in-flight order is invisible until the next scheduled cycle — arbiter/engine.py:1261-1403 (`build_engine` calls `seed_executor` + `restore_persisted_pause` but never `reconcile`/`_reconcile_pending_orders`) — why — The lane brief expects "reconcile-on-start recovers cleanly." `build_engine` seeds the sim executor and restores the pause flag, but performs NO reconciliation pass against the broker. `_reconcile_pending_orders` runs only inside `run_cycle`/`run_fast_iteration`, and the standalone `reconciler.reconcile` (reconciler.py:104) is purely diagnostic and is not wired into startup at all. So after a crash, an orphaned in-flight order (P1) or a broker/ledger divergence persists undetected until the daemon's first in-session iteration — which, for the post-close one-shot path or a relaunch outside session hours, may be a long sleep away (daemon.py:203, `_until_next_open_capped`). The grep-confirmed absence of a `reconcile(` call in `build_engine` means crash recovery is lazy, not eager. — recommended action — call `_reconcile_pending_orders(clock.now())` (alpaca_paper) once inside `build_engine` after wiring, gated on `isinstance(executor, AlpacaAdapter)`, so a relaunch immediately squares the local ledger against the broker before any new submission. Optionally run the diagnostic `reconcile()` and abort/alert on non-clean divergence.

### [P2] — SimExecutor restore loses `_daily_pl` and `_reports`; daily-PL silently resets to 0 across any relaunch — arbiter/shared/sim_executor.py:141-156 (`restore_state`) vs :116-139 (`export_state`) — why — `export_state` serializes only `cash`, `realized_pl`, and positions. `_daily_pl` (accumulated on every sim SELL, sim_executor.py:207) and `_reports` are not exported, and `restore_state` reinitializes neither — `__init__` sets `_daily_pl = 0.0` and `restore_state` never touches it. After ANY relaunch the sim account's `daily_pl` is zero even mid-session, so any breaker / report that reads `AccountSnapshot.daily_pl` (get_account, line 107) for sim is reset by a restart. `_reports` loss is benign (audit trail lives in the DB), but `_daily_pl` feeding a daily-loss breaker is a correctness gap: a relaunch resets the day's running loss to 0 and could re-arm trading past a daily-loss limit. — recommended action — either persist `_daily_pl` (with a date stamp so it resets on a true new session, not on a restart) in `export_state`/`restore_state` and the `sim_account` row, or document that sim `daily_pl` is intentionally non-durable and ensure no breaker depends on it. Confirm via grep which breaker reads `daily_pl`.

### [P3] — Daemon scheduling state is purely in-memory → relaunch can re-fire a full cycle slot or skip the daily ingest dedupe — arbiter/runtime/daemon.py:61-69 (`DaemonState`) — why — `fired_full_slots`, `last_ingest_date`, and `last_session_open` live only in the in-memory `DaemonState` (re-created empty at daemon.py:171 on every `run_daemon` entry). On a `KeepAlive` relaunch mid-session: (a) `_due_full_slots` (daemon.py:113-131) treats any already-passed slot as unfired and re-runs the FULL `run_cycle` for it (C7a "missed-slot" logic can't distinguish "missed" from "already ran then crashed") → a second entry-generating cycle for slots that already fired; (b) `last_ingest_date` resets to None, so `_run_full_cycle` re-invokes `ingest_fn` (relies on ingest itself being idempotent); (c) `last_session_open` resets, so the open→closed post-close sweep (daemon.py:198) may not fire if the relaunch lands after close. Entry re-submission is partly backstopped by order idempotency (dedup_hash), so impact is bounded, but a duplicate full cycle is wasted work and widens the P1 window. — recommended action — persist `fired_full_slots` (date-scoped) and `last_ingest_date` to a small durable table (or reuse `engine_state`), and reload them at daemon start so a relaunch does not re-fire a completed slot. Lower priority because order/ingest idempotency limits the blast radius.

### [P3] — `snapshot_executor` SAVEPOINT path calls `conn.commit()` inside the SAVEPOINT, and the rollback path does not re-raise cleanly under a parent transaction — arbiter/execution/position_store.py:64-87 — why — The wipe+rewrite is wrapped in `SAVEPOINT snapshot_op` (good — avoids "transaction within a transaction"), but after `RELEASE snapshot_op` it also issues `conn.commit()` (line 83). If a parent/engine transaction was open, that `commit()` commits the parent's other pending work as a side effect of a position snapshot — coupling the snapshot's durability to unrelated in-flight writes. The crash window here is narrow (the SAVEPOINT is atomic) so this is not a divergence source per se, but the unconditional `commit()` blurs transaction ownership. — recommended action — drop the unconditional `conn.commit()` when operating under a caller-owned transaction, or document that `snapshot_executor` is only ever called at a transaction boundary (it currently is, at the tail of run_cycle, which is why this is P3 not higher). Confirm no caller relies on the implicit commit.

## What WORKS (verified)

- **Durable pause truly restores.** `engine_state` (migration 024) singleton row is written by `_persist_paused` on both auto-pause (engine.py:583, inside `_fire_critical_alert`) and `resume()` (engine.py:249), and restored by `restore_persisted_pause` (engine.py:266-281), which `build_engine` calls last (engine.py:1401). A non-breaker broker-fatal auto-pause survives a `KeepAlive` relaunch and the engine will NOT trade until an explicit `resume()`. Both persist and restore are fail-safe (broad except, log-only) so a DB hiccup never crashes the daemon. This closes the exact hole the migration was written for. ✔
- **Snapshot atomicity (single-cycle).** `snapshot_executor` runs the DELETE+INSERT+UPSERT inside one SAVEPOINT (position_store.py:64-87), so a crash mid-snapshot cannot leave a half-written `sim_positions`. ✔ (ordering vs the ledger is the P0 problem, not atomicity.)
- **export/restore fidelity for the held book.** `export_state` filters `shares > 0` (sim_executor.py:137) so a 0-share ghost can't be persisted and resurrected; `open_position_count` matches by filtering `shares > 0` (position_store.py:120-125). cash/realized_pl/positions round-trip exactly. ✔ (gap is `_daily_pl`, see P2.)
- **alpaca_paper open_positions reads the broker, not the stale sim snapshot.** `status()` is mode-aware (engine.py:1227-1231): sim → durable snapshot count, alpaca_paper → live `get_positions()`. So the P0 divergence is sim-only; the live broker is its own durable store. ✔
- **Single-instance lock** (daemon.py:87-104 `flock`) prevents the daemon and the 18:30 one-shot from concurrently mutating the same SQLite DB — closes the concurrent-writer corruption path. ✔

## OPPORTUNITIES TO ADD

1. **Reconcile-on-start as a first-class recovery step** (addresses P1+P2): a `recover_on_start(engine)` that, for alpaca_paper, reconciles in-flight orders AND runs the diagnostic `reconcile()` and refuses to trade (or alerts) on a non-clean divergence before the first new submission.
2. **Snapshot-after-every-mutation invariant for sim** (addresses P0): make `snapshot_executor` a post-condition of any code path that fills a sim order (entry OR exit, full cycle OR fast iteration), enforced by a single helper so a future new call site can't forget it. A test asserting "sim ledger close ⇒ durable snapshot reflects it within the same iteration" would lock this.
3. **A crash-window regression test**: simulate a fast-iteration sim SELL, then rebuild the engine WITHOUT an intervening full `run_cycle`, and assert the seeded executor does NOT hold the closed position and cash/realized_pl match the durable ledger. This is the single highest-value test for this lane.
4. **Persist a "session day" stamp** alongside sim `_daily_pl` (P2) and daemon `fired_full_slots`/`last_ingest_date` (P3) in a tiny durable scheduling table so a relaunch resumes scheduling state instead of re-deriving it.
5. **Heartbeat-based liveness recovery**: the heartbeat file (daemon.py:71-84) is written but never read on start; a relaunch could inspect the last heartbeat's `now`/`open_positions` to detect a crash mid-session and trigger an eager reconcile.
