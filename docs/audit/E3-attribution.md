# E3 — Outcome Attribution Audit (sub-project #5a)

**Auditor lane:** E3 (outcome attribution correctness)
**Date:** 2026-06-19
**Scope:** `arbiter/signals/opinion_store.py`, `arbiter/evaluation/attribution.py`, `engine._persist_cycle_opinions`, plus the three resolver callers (`orchestrator/outcome_runner.py`, `execution/exit_monitor.py`, `runtime/daemon.py`). Trust/Brier math is E1's lane and was NOT re-derived.
**Method:** static read + 4 empirical probes against an in-memory migrated DB (`.venv/bin/python`); existing suite (45 tests across the 4 modules) re-run green.

## Verdict

**PASS with reservations.** The core attribution mechanics are correct: opinions are persisted PIT-clean and insert-only; the fan-out writes exactly one outcome per contributing advisor carrying that advisor's OWN stance/confidence; the realized alpha/binary are identical across advisors (stance is pure metadata, touches no price read — verified in `outcome_labeler.label`); idempotency holds across sweep / sync close-out / reconcile / stranded-retry via the per-(idea,advisor) existence guard; the strict-subset retry (`stored ⊊ linked`) correctly completes a partial fan-out instead of stranding it; the NULL-idea_id case is persisted-but-never-recovered as designed. **No P0.** The reservations below are real but narrow: they require an out-of-order fallback, a lost monitoring signal on one path, and a stance-vs-fallback masking risk that is structurally possible because the fallback emits a REAL advisor id.

## Findings

### [P1] — Fallback-first permanently strands the real stance (neutral 0.0 masks the true forecast) — `arbiter/evaluation/attribution.py:135-143` (guard) + `:91-132` (fallback) — why — The per-(idea,advisor) existence guard keys ONLY on `advisor_id`, and the fallback emits a REAL advisor id (`A1.insider`/`A1.congress`, identical to what the live advisor emits — see `engine.py:1190`, `exit_monitor`/`daemon` `_advisor_id_for`). If the neutral fallback (stance=0.0) ever writes an outcome for advisor X BEFORE X's real opinion is recoverable, the guard then blocks X's real stance forever — the advisor is scored on a permanent p_hat=0.5 instead of its actual directional forecast. Empirically confirmed (probe "Scenario C"): fallback writes `(A1.congress, stance=0.0)`; a later retry with the real opinion now persisted returns `[]` and the stance stays 0.0 — the real 0.8 forecast is stranded. — recommended action — Either (a) have the fallback emit a *distinct, reserved* advisor id (e.g. `A1.congress#proxy`) so it can never collide with the real advisor's outcome, OR (b) gate the fallback on "this idea has ZERO linked opinions AND will never get one" rather than "none recoverable right now." Today the only thing preventing this is call-ordering luck: opinions are persisted at the entry/decision cycle (`engine.py:981`) long before resolution, so in the happy path the real opinion exists first. The risk window is any idea that reaches resolution with its opinion-link lost (NULL-idea_id, a persist error counted at `engine.py:824`, or a legacy/orphan close) — exactly the cases the fallback is designed to catch.

### [P2] — Fallback fired on the daemon post-close sweep is invisible (`attribution.fallback_proxy` metric dropped) — `arbiter/runtime/daemon.py:257-261` — why — The spec's whole safety mechanism for "the fallback must not silently become the common path" is the `attribution.fallback_proxy` counter (E1). `run_outcome_sweep` defaults `metrics=None` (`outcome_runner.py:19`) and the daemon post-close call omits the `metrics=` argument entirely. Any opinion-persist regression that surfaces as a fallback on the post-close path increments NO counter and is invisible to monitoring — the exact silent-common-path failure the design warns against. (The in-cycle engine sweep at `engine.py:1192` and the exit-monitor path DO pass `metrics`; only the daemon path is blind.) — recommended action — Pass `metrics=engine._metrics` to the daemon's `run_outcome_sweep` call so fallback firings on the post-close path are counted like every other path.

### [P2] — Re-emitted opinions for an already-live idea are persisted with idea_id=NULL (un-attributable) — `arbiter/engine.py:981` + `_persist_cycle_opinions:799-817` — why — The `ideas_by_key` link index is built ONLY from `ideas` (the freshly-built ideas this cycle). That list explicitly EXCLUDES held tickers (`engine.py:950-956`) and is never merged with `active_ideas` (loaded at `engine.py:931` but unused for linking). So on every cycle AFTER the idea's creation cycle, the advisor re-emits the same opinion and it is persisted with `idea_id=NULL`. This is benign for resolution (the ORIGINAL link from the creation cycle is what `query_opinions_for_idea` recovers, and that row persists), but it (a) inflates the NULL-idea_id opinion population, and (b) means the *latest* stance the advisor holds at resolution time is NOT what gets scored — the stale creation-cycle stance is. If conviction drifts over the hold, attribution scores the entry-time forecast, not the lived one. — recommended action — Document this as intended (score the entry-time thesis), OR merge `active_ideas` into `ideas_by_key` so live-idea re-emissions link to their owning idea and the dedup-to-latest (`attribution.py:78-80`) picks the freshest stance. At minimum, add an assertion/metric on the NULL-idea_id persist rate so a link-regression (e.g. a future bucket-mapping drift) is visible.

### [P3] — Opinion idempotency guard can collapse two genuinely-distinct same-fingerprint opinions at the same as_of — `arbiter/signals/opinion_store.py:60-73` — why — The SELECT-guard dedups on `(advisor_id, idea_id, source_fingerprint, as_of)`. If a single advisor legitimately holds two opinions on the same idea at the same instant that happen to share a `source_fingerprint` (e.g. two filings collapsed to one fingerprint), the second insert is skipped and the first id returned. For the 2-advisor MVP each (advisor, idea, as_of) is 1:1 so this cannot fire today, but the guard's correctness rests on `source_fingerprint` being unique-per-distinct-opinion, which is an advisor-side invariant not enforced here. — recommended action — Note the invariant in the docstring (fingerprint MUST be distinct per distinct opinion) and consider including a stance/confidence discriminator, or asserting it, before adding advisors that can emit multiple same-fingerprint opinions per cycle.

### [P3] — Strict-subset retry depends on a (ticker, horizon_bucket) join that silently mis-links on bucket drift — `arbiter/execution/exit_monitor.py:442-454` — why — The stranded-closeout sweep matches a filled SELL row to its idea by `i.dedupe_key_ticker = o.ticker AND i.dedupe_key_bucket = o.horizon_bucket` (the SELL row carries no idea_id). The same typed (ticker, bucket) equality underpins the opinion→idea link (`engine._persist_cycle_opinions`). Both are correct ONLY while advisor `horizon_days` and idea `horizon_days` bucket identically — verified true for the MVP (form4→180→LONG, congress→90→MEDIUM; `signals/emit.py:46-47` vs `engine.py:959`). A future advisor whose emit-horizon buckets differently from the idea it informs would silently link nothing (opinion → NULL idea_id → fallback) and the SELL → wrong/no idea. — recommended action — When a third advisor or a configurable horizon lands, stamp `orders.idea_id` on the SELL (the BUY path already has the link) and replace the (ticker, bucket) join with the explicit idea_id, removing the bucket-drift dependency from both the retry selection and the opinion link.

## Verified-correct (no finding)

- **Per-advisor fan-out carries each advisor's own stance/confidence** — `attribution.py:144-159`; realized `alpha_bps`/`binary` are identical across advisors because `outcome_labeler.label` derives them purely from PIT prices and `stance_score` is metadata-only (`outcome_labeler.py:185-205`). Probe "two opinions" confirms two distinct stances, one shared alpha.
- **Idempotency across sweep / sync close-out / reconcile / retry** — the existence guard (`attribution.py:135-138`) excludes superseded rows but a `supersede_outcome` always inserts a NEW active row (`db/helpers.supersede_row`), so the guard still sees the advisor and does NOT duplicate (probe "Scenario D": 2 total rows, 1 active, no duplicate). Double-resolve returns `[]`.
- **Strict-subset stranded-retry** — `exit_monitor.py:455-466`: re-selects iff `stored ⊊ linked` (work remains) OR the zero-linked/zero-stored fallback case; a fully-resolved idea (`stored == linked`) is skipped. Matches the partial-write-then-crash recovery the docstring promises (`attribution.py:140-166` test confirms missing-advisor-only re-label).
- **NULL-idea_id handling** — `persist_opinion` accepts `idea_id=None` with an `IS NULL` idempotency branch (`opinion_store.py:60-65`); `query_opinions_for_idea` filters `idea_id = ?` so NULL rows are never recovered/attributed — persisted-for-audit-only, exactly as specced.
- **PIT-cleanliness** — no `datetime.now()` in either file; all timestamps flow from the injected clock / `as_of` / `cutoff_as_of`. Insert-only (no UPDATE/DELETE on opinions).
- **LookupError propagation** — a missing PIT bar raises out of the labeler, aborts the write loop before any state flip, and the callers (`outcome_runner.py:149`, `exit_monitor.py:342`) leave the idea MONITORED for retry; because the guard runs before the loop, the retry re-labels only the missing advisors.

## OPPORTUNITIES TO ADD

1. **Reserved proxy advisor namespace** — make the fallback emit a sentinel id (`<advisor>#proxy` or a dedicated `A0.proxy`) so a neutral fallback can NEVER collide with or mask a real advisor's outcome (closes the P1 risk structurally rather than relying on call-ordering).
2. **NULL-idea_id rate metric** — emit a per-cycle counter of opinions persisted with `idea_id=NULL`; a sudden rise is the early warning that the (ticker, bucket) link has drifted/broken (the P2/P3 silent-failure modes), complementing `attribution.fallback_proxy`.
3. **Fallback-rate alert threshold** — `attribution.fallback_proxy` is recorded but (per the daemon gap) not on every path and has no alert. Wire it through ALL three resolver callers and alert when the fallback fraction of resolutions exceeds a small threshold — the spec's stated intent ("can't silently become the common path") needs an actual alarm, not just a counter.
4. **Test: fallback-first-then-real-opinion** — the suite covers partial-fanout-then-retry and no-opinion-fallback, but NOT the Scenario-C ordering (fallback fires, real opinion arrives later). Add a regression test asserting the real stance is NOT masked, so the P1 fix is locked in.
5. **Stamp `orders.idea_id` on SELL rows** — removes the (ticker, bucket) join from the stranded-retry and lets attribution survive multi-advisor / configurable-horizon expansion without the bucket-drift fragility (P3).
