# D2 — Fill Confirmation & Reconciliation Audit

**Lane:** D2 — pending→filled lifecycle & reconciliation
**Auditor:** READ-ONLY (no source/test/config modified)
**Date:** 2026-06-19
**Scope:** `engine._reconcile_pending_orders` / `_mark_order_terminal` / `_close_out_filled_sell`, `execution/alpaca_adapter.py::get_order`/`place`, `execution/reconciler.py`, `_bound_submit`.

## Verdict

**CONDITIONAL PASS.** The core lifecycle is sound: only a real broker `filled` advances an idea → MONITORED, terminal states are written to a non-`pending` local status so they stop being re-queried, and the `_bound_submit` filled-vs-pending split correctly defers the advance to the next cycle's reconcile. Two correctness gaps stop this being a clean pass: (1) the BUY idea-advance still uses the fragile `(ticker,bucket)` join instead of the `orders.idea_id` link that exists precisely to replace it — the documented multi-idea hazard re-opens here and can advance the WRONG idea; (2) a **partial** local order status is **terminal in the schema selection** — partials are never re-selected as `pending`, so the residual fill is never reconciled and the idea can be stuck-partial-forever / under-advanced. Several P2/P3 robustness gaps (lost-response surfacing, expired-buy idea, `place()` partial mislabel) are noted below.

---

## Findings

### [P1] — BUY idea-advance ignores `orders.idea_id`, re-opening the fragile (ticker,bucket) join — `arbiter/engine.py:357-364` — why — Migration `023_orders_idea_id.sql` added `orders.idea_id` expressly so the lifecycle can "tie an order to its owning idea EXACTLY, instead of relying on the (ticker, horizon_bucket) join (fragile if the one-live-bucket-per-held-ticker invariant ever breaks)", and `_bound_submit` (engine.py:1094-1107) populates it at submit time. But the reconcile BUY branch resolves the idea with `SELECT ... WHERE dedupe_key_ticker=? AND dedupe_key_bucket=? AND state != MONITORED ORDER BY created_at DESC LIMIT 1` — it never reads `row["idea_id"]`. If two ideas ever share the same (ticker,bucket) (supersede churn, a re-opened idea, or any break of the one-live-bucket invariant), the fill can advance a *different* idea than the one the order was actually placed for, while the true idea stays pre-MONITORED. This is the exact failure 023 was written to prevent — recommended action: when `row["idea_id"]` is non-NULL, advance that idea by id (load its state, advance only if not already MONITORED); fall back to the (ticker,bucket) join only for legacy NULL rows.

### [P1] — Partial fills are never reconciled to completion: `partial` is excluded from the `pending` re-query, so the residual fill is dropped and a partial BUY under-advances / a partial SELL leaks shares — `arbiter/engine.py:302,330-333` + `reconciler.py:80` — why — `_reconcile_pending_orders` re-queries only `WHERE status='pending'`. When `get_order` reports `partial`, the row is written `status='partial'` (engine.py:333-339) and on the next cycle is **no longer selected**, so the order is never re-checked even though the broker can still fill the remainder. Consequences: (a) a partial BUY advances its idea to MONITORED on the *partial* (engine.py:354-372, the else-branch runs for any non-terminal `partial`), so the idea is treated as fully entered while shares are still outstanding — and the residual fill is never seen; (b) a partial SELL is intentionally left for "the next cycle to re-sell the residual" (engine.py:455-469 / docstring), but there is no mechanism that re-selects the `partial` row or re-emits a residual SELL, so the residual is stranded and `reconciler._local_positions` (only sums `status='filled'`, reconciler.py:80) will also miss it, producing a silent LOCAL/BROKER qty divergence. Recommended action: keep partials in the reconcile working set (e.g. `WHERE status IN ('pending','partial')`), and either advance the BUY idea only on a *full* fill or track filled_qty vs requested qty so the residual is reconciled; confirm a concrete residual-SELL re-emission path exists for partial sells.

### [P2] — Filled-at-broker-but-lost-response order is not surfaced as a divergence in the normal flow — `arbiter/execution/alpaca_adapter.py:177-192,250-266` — why — On a `place()` POST whose response is lost (BrokerError after retries) the adapter returns `status="rejected"` with the original `client_order_id`; `submit_order` (submit.py:309-331) then **raises BrokerError and does NOT persist the order row**. If Alpaca actually accepted/filled that order (the lost-response case the idempotent `client_order_id` is designed for), there is now a broker fill with **no local `pending` row** — so `_reconcile_pending_orders` (which only scans local `pending` rows) will never query it, and the idea stays pre-MONITORED forever. The position-level `reconciler.reconcile` (reconciler.py:104) is the only thing that would catch it as a `BROKER_ONLY` divergence, but per the call-site comments it is a diagnostic tool not wired into the live cycle (reconciler.py:18-20). Net: a lost-response fill is detectable only if someone runs the position reconciler; it is not surfaced by the pending→filled lane itself. Recommended action: on a lost-response BrokerError, persist the order row as `pending` (idempotent `client_order_id` makes a later `get_order` safe) so reconciliation can resolve it, OR wire `reconciler.reconcile` into the live cycle and alert on `BROKER_ONLY`/`LOCAL_ONLY`.

### [P2] — Expired/cancelled BUY silently abandons its idea with no alert or re-attempt — `arbiter/engine.py:402-448` — why — `_mark_order_terminal` correctly maps broker `expired`/`canceled`/`rejected` to a terminal local status so the row stops being re-queried (good — no stuck-pending here), and deliberately does NOT advance the idea. But a `day` BUY that expires unfilled at market close leaves its idea permanently pre-MONITORED with only an info/audit log — no re-submission next session and no operator alert. The idea is effectively dropped on the floor. This is arguably intended ("stays pre-MONITORED for a later re-attempt"), but no mechanism re-attempts it. Recommended action: confirm whether an expired BUY's idea should be re-queued for next-session submission; if so, reset/flag the idea; if not, emit an alert so the dropped entry is visible rather than silent.

### [P2] — `place()` labels any positive fill as fully `filled`, masking a same-call partial — `arbiter/execution/alpaca_adapter.py:194-204` — why — `place()` sets `status="filled" if fill_qty > 0 else "pending"` — it does NOT compare `filled_qty` against requested `qty` the way `get_order` does (alpaca_adapter.py:280-283). A market order that fills only part of its quantity in the immediate POST response is recorded `filled` (and the idea advanced immediately in `_bound_submit`), with the residual then invisible to reconciliation (it is no longer `pending`). Lower likelihood for liquid equities at paper, but it is an inconsistency with `get_order` and compounds the P1 partial gap. Recommended action: mirror the `get_order` full-vs-partial logic in `place()` (`filled` only when `filled_qty >= requested qty`, else `partial`).

### [P3] — Reconcile idea-advance is silent when no matching idea row is found — `arbiter/engine.py:365` — why — The BUY branch advances only `if idea_row is not None`; if the join returns nothing (idea superseded/closed/missing, or the bucket value mismatches), the order is marked `filled` but **no idea is advanced and nothing is logged about the miss**. Combined with P1 this means a join failure is indistinguishable from a normal no-op. Recommended action: log a warning/audit when a filled BUY reconciles but resolves no advanceable idea.

### [P3] — Double-advance is prevented only by the `state != MONITORED` filter, not by an order-level guard — `arbiter/engine.py:330-339,362-363` — why — "advance exactly once" relies on two things lining up: the order row flips `pending → filled` before the idea lookup, and the idea SELECT excludes `state = MONITORED`. This holds for the common path, but it is defense-by-query-filter rather than an explicit idempotency guard, and it interacts badly with the P1 join (a *second* live idea in the same bucket that is not yet MONITORED would be advanced on a fill meant for the first). Recommended action: once P1 is fixed to advance by `idea_id`, gate the advance on the order not already being terminal/filled (the UPDATE at :336 already gives single-flip semantics per order) and keep the `state != MONITORED` guard as a backstop.

### [P3] — `get_order`/`cancel`/lost-response reports hardcode `side=OrderSide.BUY` — `arbiter/execution/alpaca_adapter.py:257,290` — why — `get_order` always returns `side=OrderSide.BUY` regardless of the broker payload. The reconcile path is aware of this and correctly branches on the **local** `row["side"]` (engine.py:348-351, with an explicit comment), so SELL reconciliation is not broken today. But any future consumer that trusts `report.side` from `get_order` will silently mis-handle SELLs. Recommended action: populate `side` from the broker `data["side"]` in `get_order` so the report is self-consistent; keep the engine's local-side branch as the source of truth.

## OPPORTUNITIES TO ADD

- **Wire `reconciler.reconcile` into the live cycle (or a periodic sweep) with alerting.** It already computes LOCAL_ONLY / BROKER_ONLY / QTY_MISMATCH divergences but is diagnostic-only (reconciler.py:18-20). Running it each session would catch the lost-response fill (P2), the stranded partial (P1), and any drift the pending-lane can't see — turning silent divergences into alerts.
- **Persist `filled_qty` / `avg_fill_price` / `requested_qty` on the order row** so partial reconciliation has the state it needs to compute a residual, and so `reconciler._local_positions` can net partials rather than only summing `status='filled'`.
- **Cover Alpaca `new`/`accepted`/`pending_new`/`pending_cancel`/`done_for_day` explicitly in `get_order`'s status map.** They currently fall through to `pending` via the `else` (alpaca_adapter.py:284-285), which is safe but undocumented; an explicit mapping prevents a future Alpaca status (e.g. `held`, `stopped`) from being silently treated as pending forever.
- **Add a stuck-pending watchdog:** alert when an order has been `pending` across N reconcile cycles / M hours, so a genuinely stuck order (broker state never reached terminal/filled) is surfaced rather than re-queried indefinitely.
- **Idempotency test for the advance-by-idea_id path** once P1 is fixed: two ideas sharing (ticker,bucket), assert the fill advances exactly the order's `idea_id` and the other idea is untouched.
