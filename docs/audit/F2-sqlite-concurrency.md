# F2 — SQLite Concurrency & Connection Handling — Audit

**Lane:** F2 (READ-ONLY audit)
**Date:** 2026-06-19
**Auditor scope:** SQLite usage correctness under concurrency — `db/connection.py`, the per-thread advisor connections (`engine._build_a1_*_fn`), SAVEPOINT usage in `db/helpers.py` and `execution/position_store.py`, daemon long-lived conn vs advisor threads, busy_timeout/locking, `INSERT OR …` paths.

## Verdict

**PASS with reservations.** The core concurrency contract is honoured: no `sqlite3.Connection` object is shared across threads (each advisor thread opens its own fresh connection and closes it in a `finally`), WAL is enabled, and all multi-statement mutations are wrapped in `SAVEPOINT`/`RELEASE` so they are safe whether or not a parent transaction is open. The advisor threads are strictly **readers** (`detect_signals` only `.fetchall()`s), and WAL lets readers proceed without blocking the single writer. The one real exposure is **no explicit `busy_timeout` is set** — the code relies on Python's *implicit* 5000 ms default, which is undocumented at the call site and one `sqlite3` version / config change away from becoming 0 ms (instant `database is locked`). No P0. One P1 (implicit-only busy_timeout) and several P2/P3 hardening items.

### Architecture confirmed
- `engine.conn` is a single long-lived connection created at `build_engine` (engine.py:1313) and used by the daemon's main loop for all writes (`run_cycle`, `run_fast_iteration`, `snapshot_executor`, `idea_store`, `outcome_runner`).
- Inside `run_cycle` (engine.py:914) the main thread fans out advisor callables via `run_named_advisors_parallel` → `ThreadPoolExecutor` (scheduler.py:64). Each advisor opens its **own** connection (engine.py:144 / :174) and closes it in `finally`. SQLite objects are therefore never touched from two threads → the `ProgrammingError: SQLite objects created in a thread can only be used in that same thread` class of bug is avoided. `check_same_thread` is left at its default (`True`), which is the correct guard given this pattern.
- Single-instance `flock` (daemon.py:87) prevents two *processes* (daemon + 18:30 one-shot) from writing the same DB. Good — that closes the cross-process writer-vs-writer race.

## Findings

`[P1] — busy_timeout is implicit-only; never set via PRAGMA — arbiter/db/connection.py:35-43 — ` The factory enables WAL + FK but **never executes `PRAGMA busy_timeout`**. Under concurrency the daemon's main `engine.conn` holds the single write transaction across a `run_cycle`, while up to 8 advisor reader threads (scheduler.py:25, each its own conn) hit the same DB file. WAL lets readers proceed against the last checkpoint, but a reader that needs to *start* while a checkpoint or a brief exclusive moment is in flight, or any future writer-side contention, will get `sqlite3.OperationalError: database is locked` unless a busy handler waits. Today it "works" only because CPython's `sqlite3` ships a **default `busy_timeout` of 5000 ms** (verified: `PRAGMA busy_timeout` → 5000 on this interpreter). That default is not part of the SQLite C contract (raw `sqlite3_open` defaults to 0), is invisible at the call site, and is exactly the kind of thing a Python upgrade or a `connect(timeout=…)` refactor silently changes. **Recommended action:** make it explicit and generous in `get_connection`, e.g. `conn.execute("PRAGMA busy_timeout=5000;")` (or higher) right beside the WAL/FK pragmas, so the locking-wait behaviour is owned by this code, not by an interpreter default.

`[P2] — Long-lived write transaction spans the parallel advisor fan-out — arbiter/engine.py:914 (within run_cycle, conn writes at 255/335/416/1107/1131/1175…) — ` `engine.conn` runs in legacy autocommit-off mode (`isolation_level=''`, verified), so the first DML in a cycle silently *opens* a transaction that stays open until the next `conn.commit()`. The advisor reader threads are spawned at line 914, potentially while such a write transaction is mid-flight. With WAL + a busy_timeout this is tolerable (readers see the pre-transaction snapshot; the writer never blocks readers), but a long-open writer transaction also **defers WAL checkpointing**, so the `-wal` file can grow unbounded across a long daemon session and the auto-checkpoint can briefly contend. **Recommended action:** keep write transactions short — commit (as the code mostly does) immediately after each logical mutation rather than letting one implicit transaction straddle the advisor fan-out and the exit monitor; consider a periodic `PRAGMA wal_checkpoint(TRUNCATE)` on the daemon's idle (market-closed) branch (daemon.py:195-203) to bound `-wal` growth.

`[P2] — No explicit busy_timeout on the per-advisor reader connections either — arbiter/engine.py:144,174 — ` Same root cause as the P1: the advisor threads call `get_connection(db_path)`, so fixing the factory fixes them too. Called out separately because these are the connections most likely to *hit* a lock (they start fresh, mid-cycle, against an actively-written DB). The `try/finally` close is correct; the missing wait is the gap. **Recommended action:** covered by the P1 factory fix — no separate change needed once `busy_timeout` is set in `get_connection`.

`[P3] — SAVEPOINT error path issues bare ROLLBACK-to + RELEASE without re-RELEASE safety, but is correct — arbiter/db/helpers.py:129-141, :169-182; arbiter/execution/position_store.py:64-87 — ` The pattern `SAVEPOINT x` / … / `RELEASE x` / `commit()`, with `except: ROLLBACK TO x; RELEASE x; raise` is correct and *is* safe whether or not a parent txn is open (the stated design goal — verified). One subtlety: if `_insert_row_no_commit` raises *before* the `SAVEPOINT` line executes, there is no savepoint to roll back — but here the `SAVEPOINT` is the first statement in the `try`-guarded block's lead-in, so that ordering holds. The `commit()` after `RELEASE` is belt-and-suspenders (RELEASE of the outermost savepoint when no parent txn exists already commits); harmless. **Recommended action:** none required; optionally add a comment that the trailing `commit()` is a no-op when the savepoint was the outermost frame, to prevent a future reader "simplifying" it into a double-commit bug.

`[P3] — INSERT OR IGNORE re-read race is handled, but relies on same-conn read-after-write — arbiter/ingest/identity/resolver.py:135-158 — ` `INSERT OR IGNORE` then a re-`SELECT` to fetch the winner is the right idempotent pattern. This path runs on the **ingest writer's** connection (the daemon's single conn or the one-shot's), never on an advisor reader thread, and the single-instance `flock` means no *other process* is writing concurrently — so the "two concurrent writers" comment at line 152 cannot actually occur in the current deployment (single writer by construction). The re-read is harmless defensive code. **Recommended action:** none; the comment slightly overstates the live risk (there is exactly one writer), but the code is correct and future-proof.

`[P3] — INSERT OR REPLACE on breaker_state could break FK children if any are ever added — arbiter/safety/breakers.py:115-126 — ` The breaker writer uses `INSERT … ON CONFLICT(breaker_name) DO UPDATE` (not a raw `INSERT OR REPLACE`, despite the docstring wording at :115/:117), so it does **not** delete-and-reinsert the row and therefore won't cascade-delete or orphan any future child rows. Flagged only because the docstring says "INSERT OR REPLACE" while the SQL is the safer upsert — keep the SQL, fix the doc. With FK enforcement on (connection.py:40), a literal `INSERT OR REPLACE` here would be a latent hazard if `breaker_state` ever gains children. **Recommended action:** correct the docstring to "upsert via ON CONFLICT … DO UPDATE"; do not switch to `INSERT OR REPLACE`.

`[P3] — get_connection does not set check_same_thread or document the no-share contract — arbiter/db/connection.py:35 — ` The thread-safety of the whole design rests on the convention "one connection per thread, never shared." That convention is enforced only by `check_same_thread=True` (the default) raising at runtime if violated, and by the advisor builders' discipline. **Recommended action:** add a one-line docstring note in `get_connection` that connections are NOT thread-shareable and must be created per-thread (the engine builders already do this); this makes the contract explicit for future lanes that might be tempted to pass `engine.conn` into a worker.

## OPPORTUNITIES TO ADD

- **Explicit `PRAGMA busy_timeout` in the factory** (resolves P1 + P2-reader in one line) — single highest-value change; removes the dependency on an interpreter default.
- **Periodic `wal_checkpoint(TRUNCATE)`** on the daemon's market-closed branch (daemon.py:203) to bound `-wal`/`-shm` file growth over multi-day runs.
- **A concurrency regression test**: spawn the real `ThreadPoolExecutor` advisor fan-out against a temp file-backed (not `:memory:`) DB while the main conn holds an open write transaction, and assert no `database is locked` is raised. `:memory:` DBs cannot reproduce WAL/locking semantics, so the existing tests likely don't cover this — a file-backed stress test would lock in the busy_timeout contract.
- **Consider `PRAGMA synchronous=NORMAL`** (safe and standard under WAL) in the factory if checkpoint/commit latency ever matters for the daemon's fast loop; document the durability trade-off if adopted.
- **Centralise all pragmas** (WAL, FK, busy_timeout, synchronous) in `get_connection` so every connection — main and per-thread — is provably configured identically; today only WAL+FK are guaranteed, and any code path that constructs a connection another way would silently differ (currently there is only one `sqlite3.connect`, so this is preventative).
