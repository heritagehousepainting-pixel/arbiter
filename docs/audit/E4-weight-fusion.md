# E4 — Weight Resolver & Fusion Audit

**Auditor lane:** E4 (weight resolver & fusion — how learned weights become a fused conviction)
**Date:** 2026-06-19
**Scope (read-only):** `arbiter/trust/weight_resolver.py`, `arbiter/fusion/engine.py`, `arbiter/fusion/pool.py`, `arbiter/fusion/correlation.py`, `arbiter/fusion/dedup.py`; call sites in `arbiter/engine.py` and `arbiter/orchestrator/cycle.py`. Trust math (E1) and attribution (E3) trusted, not re-audited.
**Orientation:** Learning-loop spec D1/D3/D6 (per-advisor floor/graduate/suppress), D4 (correlation v1=ρ0), INTERFACES.md §5 (caps: ceiling 0.50, floor 0.02, negative-skill→0).

**VERDICT: PASS WITH FINDINGS.** The cold-floor bootstrap cannot deadlock (a single cold advisor trades at the floor, non-shadow; floor 0.25 < ceiling 0.50 so a cold advisor never outranks a graduated one 1-vs-1). The Phase-1 ρ=0 default is safe (effective_n≈N, no spurious deflation). Single-advisor weight is well-behaved (effective_n=1.0, diversity_factor=1.0). However there is **one real normalization/double-count defect (P1, currently latent), one architectural cross-ticker fusion hazard (P1), and several P2/P3 correctness and documentation gaps.** No P0.

## FINDINGS

### [P1] — Same-advisor / multiple-run_group opinions collide in pool normalization (double-counts signal, corrupts contributions) — `arbiter/fusion/pool.py:74-123`
**Why:** `pool_opinions` keys `raw_weights`, `norm_weights`, and `advisor_contributions` by `op.advisor_id`. `dedup_bucket` only merges opinions sharing the SAME `(advisor_id, run_group_id)` — so one advisor emitting two opinions with DIFFERENT run_group_ids in the same bucket passes both through to the pool. The signal loop (`for op in opinions: ... signal_strength += contrib`) then iterates BOTH opinions and adds both contributions to `signal_strength`, but each writes to the same `advisor_contributions[advisor_id]` key, so the dict retains only the LAST one. Reproduced: advisor `a` with +1.0 (rg1) and −1.0 (rg2) plus advisor `b` +1.0 (rg3), equal weights → `signal_strength = 0.5`, but `advisor_contributions = {a: -0.5, b: 0.5}` (sums to 0.0 ≠ 0.5). `norm_weights` also treats `a` as a single 0.5 participant while it injected two terms. The reported conviction, the attribution surface (E3 consumes `advisor_contributions`), and `n_pool`/effective_n all disagree about how many times the advisor was counted.
**Reachability:** Currently LATENT. The only live advisors (MiroFish `adapter.py:133`, generic `emit.py:124`) assign ONE `run_group_id` per advisor invocation, and the MiroFish SHORT+MEDIUM split lands in different buckets — so today no advisor produces two run_groups in one bucket. It becomes live the moment (a) an advisor is invoked twice per cycle, (b) an advisor emits per-ticker run_groups (likely once cross-ticker pooling, see next finding, is intentional), or (c) a future adapter splits one advisor's run into multiple run_groups within a horizon.
**Recommended action:** Make pool.py robust to repeated advisor_ids: either accumulate per (advisor_id, run_group_id) like dedup, or accumulate `advisor_contributions` additively (`+=`) and base `norm_weights`/effective_n on the actual count of pooled opinions, not the distinct-advisor set. Add a regression test feeding two same-advisor different-run_group opinions into one bucket and asserting `sum(advisor_contributions.values()) == signal_strength`.

### [P1] — Cross-ticker bucket pooling: one fused conviction is computed across ALL tickers in a bucket and reused for every idea — `arbiter/orchestrator/cycle.py:242-261`, `arbiter/engine.py:1001-1015`
**Why:** `run_cycle` groups opinions purely by `op.horizon_bucket` (`opinions_by_bucket`) with NO ticker key, then for each idea calls `fuse(bucket_opinions, bucket)` using the entire bucket's opinion list regardless of ticker. `_bound_fuse(opinions, bucket)` in `engine.py` also has no ticker parameter. Consequently a SHORT opinion about ticker X and a SHORT opinion about ticker Y are log-pooled together, and the single resulting conviction is handed to `decide` for EVERY pending idea in that bucket. The risk: an advisor's bullish call on AAA leaks into the conviction used to size a BBB order; a lone-bull "dissenter" on a totally unrelated ticker can apply/suppress the −0.10 tax on this idea; effective_n/dispersion mix unrelated names. This is acceptable only if the deployment guarantees one ticker per bucket per cycle (single-name universe). With multiple tickers it silently cross-contaminates convictions and breaks per-idea attribution.
**Recommended action:** Key the fusion group by `(ticker, bucket)` (or pass `ticker` into `fuse`/`_bound_fuse` and filter opinions to that ticker). If cross-ticker pooling is intentional for a thesis-level signal, document it explicitly and gate it; do not let it be an accident of the grouping key. At minimum assert/guard that all pooled opinions share the idea's ticker.

### [P2] — Two cold advisors at the floor tie a single graduated max-trust advisor — `arbiter/trust/weight_resolver.py:34-44`
**Why:** The module docstring justifies floor=0.25 < ceiling=0.50 by saying it prevents the failure mode "≥2 cold advisors outvote one proven." Verified: with two cold (0.25 each) + one graduated at the 0.50 ceiling, normalized weights are {cold 0.25, cold 0.25, grad 0.50} → the cold bloc sums to exactly **0.50 = the graduated advisor's share**. So 0.25 reduces the cold bloc from "outvoting" to "tying" a single best-proven advisor, and ≥3 cold advisors (0.75 bloc) still outvote one graduated advisor. The 1-vs-1 dominance claim ("a COLD advisor always gets a smaller pool share than a fully-graduated max-trust advisor") is correct; the multi-cold claim is overstated.
**Recommended action:** Either tighten the docstring to state the true guarantee (cold < graduated only pairwise; a cold *bloc* can still match/exceed one graduated advisor), or lower the floor / cap the aggregate cold share so a bloc of probationary advisors cannot match a proven one. Add a test pinning the 2-cold-vs-1-graduated normalized split so the chosen behavior is intentional.

### [P2] — Post-graduation all-suppressed bucket goes dark silently (indistinguishable from "no opinions") — `arbiter/fusion/engine.py:122-124`, `arbiter/fusion/pool.py:86-88`, `weight_resolver.py:109-118`
**Why:** The no-deadlock guarantee is explicitly scoped to *bootstrap*: cold advisors get the non-shadow floor so the bucket always has ≥1 participant. But if every live advisor is flipped to `negative_skill` (all shadow=True, weight=0 — verified the resolver produces an all-shadow bundle), `pool_opinions` returns `({}, 0.0)`, the engine hits `if not norm_weights: continue` and the bucket is simply absent from the result. `_bound_fuse` then returns a conviction=0.0 FusionOutput that is indistinguishable from "no opinions arrived." There is no explicit HALT/log signalling "all advisors muted." A genuinely all-negative-skill roster should be an observable degraded/halt state, not a silent zero.
**Recommended action:** Emit a distinct log/metric (or a FusionOutput flag) when a bucket had opinions but every contributing advisor was shadow/zero. Surface it to the quorum/health path so "all advisors suppressed" is operationally visible rather than masquerading as no-signal.

### [P2] — Docstring claims "log-opinion pool / geometric mixture" but the code is a linear weighted arithmetic mean — `arbiter/fusion/pool.py:1-33, 95-121`
**Why:** The pool docstring describes a log-opinion pool ("geometric mixture of the individual predictive distributions"). The implementation computes `signal_strength = Σ w_i_norm * stance_i` — a plain weighted ARITHMETIC mean of stances after simplex normalization. For Phase-1 stances in [-1,1] (not probabilities) the arithmetic mean is a defensible choice, so this is not a numeric bug today, but the normalization is linear, not log-pool. The mismatch will mislead the Phase-3 implementer the docstring explicitly addresses ("do NOT pre-normalise weights ... pool.py normalises"): a true log-pool would normalize in log-space / on probabilities, not on raw stances.
**Recommended action:** Rewrite the docstring to describe the actual operation (weighted-arithmetic-mean of stance scores with simplex weights) and state clearly where/whether a true log-pool is intended to arrive (calibration to p_i ∈ [0,1] first, then geometric pooling) so Phase-3 doesn't inherit a wrong mental model.

### [P3] — `advisor_contributions` are post-normalization (w_i_norm * stance), not the docstring's "before normalisation" — `arbiter/fusion/pool.py:30-32, 119-120`
**Why:** Header docstring (line 31) says contributions are "w_i * stance_i **before** normalisation," but the code stores `contrib = w_norm * calibrated` (post-normalization simplex weight). Minor, but E3 attribution reads this dict and the documented contract is wrong; a consumer expecting pre-normalization magnitudes would mis-attribute.
**Recommended action:** Fix the docstring to "normalised simplex weight × calibrated signal" (matching the later, correct docstring at lines 65-67), or expose both pre- and post-normalization contributions if E3 needs the raw form.

### [P3] — Pool's "advisor not in bundle → weight 1.0" fallback can dominate the simplex — `arbiter/fusion/pool.py:76-79`
**Why:** An opinion whose `advisor_id` is missing from the WeightBundle is given raw weight 1.0. Since cold advisors are floored to 0.25 and graduated advisors are often <1.0, a *missing* advisor silently receives the LARGEST raw weight in the pool and dominates normalization. The resolver (`resolve_weight_bundle`) keys the bundle by every live id, so in the normal path no advisor is missing — but any opinion from an id not in `live_ids` (e.g. a stale/late opinion, or a resolver/roster mismatch) would get this oversized fallback instead of being floored or excluded.
**Recommended action:** Make the fallback conservative (floor weight, or exclude unknown advisors) and log when an opinion's advisor is absent from the bundle, so a roster/bundle mismatch is visible rather than silently up-weighting an unknown source.

### [P3] — `dispersion` divides nothing but assumes norm_weights sum to 1 over pooled ops — `arbiter/fusion/correlation.py:205-213`
**Why:** Weighted variance uses `norm_weights.get(op.advisor_id, 0.0)`. This is correct only when the pooled opinion set is exactly the advisor set that norm_weights was built over. Under the same-advisor-collision case (P1), `norm_weights` sums to 1 over DISTINCT advisors while `pooled_opinions` contains more entries → the variance weights no longer sum to 1 and dispersion is understated. Not independently exploitable, but it inherits the P1 defect.
**Recommended action:** Fix as part of P1 (consistent per-opinion weighting), then assert `abs(sum(weights used) - 1.0) < eps` in dispersion under test.

## OPPORTUNITIES TO ADD

- **Aggregate cold-share cap (D3 hardening):** add a config knob capping the *total* normalized weight assignable to cold/probationary advisors (e.g. ≤ one graduated advisor's share), closing the multi-cold-bloc gap (P2) deterministically instead of relying on the per-advisor floor value.
- **Ticker-scoped fusion contract:** give `fuse`/`_bound_fuse` an explicit `ticker` parameter and have it reject (or assert) opinions for other tickers, turning the cross-ticker hazard (P1) into a compile-time/contract guarantee.
- **Invariant test: contributions reconcile to signal.** Property test asserting `abs(sum(advisor_contributions.values()) - signal_strength) < 1e-9` across random opinion sets (incl. repeated advisor_ids and run_groups) — would have caught the P1 collision.
- **"All-suppressed" health signal:** a `FusionOutput.all_suppressed` flag (or metric) feeding the quorum/HALTED path so a fully negative-skill roster is an explicit degraded state (P2), not silent zero-conviction.
- **effective_n bounds test for N=1 and identical advisors:** pin effective_n=1.0 for a single advisor and effective_n→1.0 as ρ→1.0 across all pairs (perfect correlation collapses to one effective vote), guarding the diversity_factor against future ρ-matrix wiring (D4 Phase-5).
- **Negative/out-of-range ρ guard:** `effective_n` already guards `double_sum <= 0` but does not bound ρ to [-1,1]; a malformed Lane-11 matrix with ρ>1 or ρ<−1 could produce nonsensical effective_n. Clamp ρ on read.
