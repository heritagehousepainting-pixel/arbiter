# H1 — engine.py size, responsibility creep & coupling

**Auditor lane:** H1 (READ-ONLY). **Date:** 2026-06-19. **Target:** `arbiter/arbiter/engine.py` (1403 lines).
**Orientation:** `arbiter/INTERFACES.md` (§9 execution, §10 storage, §10b foundation deviations — the DI/composition-root contract).
**Scope note:** contract drift is H2's lane; this lane is maintainability of the central engine module only.

## Verdict

**AMBER, trending RED.** The composition-root *discipline* still holds — lane modules are not importing each other; everything is composed in `build_engine`, and the heavy lifting (fuse, decide, exit_monitor, outcome_runner, trust ledger, idea_store) lives in the right lanes and is injected/called, not re-implemented. That is the load-bearing invariant and it is intact.

What has eroded is `engine.py` itself. It has quietly become a **god-object orchestrator**: one 1403-line module owns the cycle, the fast-iteration loop, the broker reconciler, the sell close-out lifecycle, the learning step, opinion persistence, the (ticker,bucket)→idea join, the market-hours heuristic, durable pause persistence, status, and the factory. `run_cycle` alone is **369 lines with 9 nested closures**. Three pieces of business logic are physically duplicated (the `advisor_id_for` horizon proxy ×4 incl. daemon; the (ticker,bucket) join SQL ×3; the reconcile+fail-closed equity block ×2). None of this is yet a *correctness* bug, but it is the exact surface where the next correctness bug will hide, and it is actively slowing every change. The P0/P1 items below are velocity/correctness-risk, not live defects.

Measured surface:
- 1403 lines; 39 imports (32 from `arbiter.*`); **14 lazy in-method imports** (`noqa: PLC0415`).
- Longest methods: `run_cycle` 369 (lines 843–1211), `build_engine` 143, `_reconcile_pending_orders` 118, `_build_learning_inputs` 96.
- 8 inline `self.conn.execute(...)` raw-SQL sites embedded in the orchestrator (engine reaches *past* the L2 db helpers and L13 idea_store straight into SQL).

## Findings

### [P0] — `run_cycle` is a 369-line god-method with 9 nested closures — engine.py:843-1211 — why: it interleaves reconcile, account fail-closed, market-hours warn, advisor gather, exit monitor, signal→idea building, opinion persistence, learning-input assembly, four bound closures (`_bound_fuse`/`_gate_callable`/`_adv_provider`/`_bound_decide`/`_bound_submit`/`_opinion_provider_map`/`_on_new_idea`/`_on_transition`), the orchestrator call, broker-fatal handling, metrics, snapshot, and the outcome sweep. No single reader can hold it; the ordering constraints (e.g. "outcome sweep MUST stay last for no-look-ahead, R6") are only protected by comments, not structure. — recommended refactor: extract the closures into named bound methods or a small `CycleContext`/`CyclePlan` object (`_gather_opinions`, `_build_ideas_from_signals`, `_make_bound_decide`, `_make_bound_submit`), and split `run_cycle` into a short ~40-line conductor calling `_preflight()` → `_entries()` → `_persist_and_learn()` → `_finalize()` (snapshot+sweep). Each step independently testable; the R6 ordering becomes an explicit method sequence, not a comment.

### [P0] — `advisor_id_for` horizon proxy is hard-duplicated 4× — engine.py:474, 534, 1190 (+ runtime/daemon.py:255) — why: `"A1.insider" if idea.horizon_days >= 180 else "A1.congress"` is copy-pasted in `_close_out_filled_sell`, `_run_exit_monitor`, the `run_cycle` outcome sweep, AND the daemon. This is the attribution key feeding the trust ledger (§5) and outcome labeler (§6); if the advisor roster ever changes (A2/A3 wired) every copy silently mis-attributes alpha to the wrong advisor and corrupts trust weights. Four edit sites, zero shared definition. — recommended refactor: define ONE `advisor_id_for(idea) -> str` (module-level function or a small `AdvisorAttribution` injected at build time, ideally derived from the same source as the `build_engine` advisor_map + the 180/90 horizon mapping at line 959). Pass it once; delete the other three + the daemon copy.

### [P1] — The (ticker, horizon_bucket) → idea join is inlined as raw SQL 3× — engine.py:358-364, 1096-1101 (+ the documented fallback join in exit_monitor/outcome paths) — why: the same `SELECT ... FROM ideas WHERE is_superseded = 0 AND dedupe_key_ticker = ? AND dedupe_key_bucket = ?` is hand-written in `_reconcile_pending_orders` and `_bound_submit`, with subtly different extra clauses (one excludes `state = MONITORED`, one orders by created_at). `idea_store.py` owns ideas and already has `load_active_ideas`/`load_ideas_by_state`/`_row_to_idea` but NO "resolve idea for (ticker,bucket)" helper — so the engine reaches past its lane into SQL. INTERFACES §10 explicitly anticipates this join (`orders.idea_id` was added to *replace* it); divergent copies risk one path resolving a different idea than another. — recommended refactor: add `idea_store.resolve_idea_for(conn, ticker, bucket, *, exclude_state=None)` (L13) and call it from both engine sites. Engine stops embedding ideas-table SQL.

### [P1] — `run_cycle` and `run_fast_iteration` duplicate the reconcile + fail-closed-equity preamble — engine.py:665-682 vs 873-892 — why: both compute `_is_adapter`, conditionally call `_reconcile_pending_orders(now)`, read `get_account()`, and bail on `equity <= 0` with near-identical `log.critical` blocks. `_safety_gate` was correctly extracted (C5) to keep the two entrypoints from drifting — this adjacent preamble was NOT, so the two daemon entrypoints can still diverge on the fail-closed posture. — recommended refactor: extract `_preflight_broker(now) -> CycleResult | None` (reconcile + equity fail-closed) and call it from both, mirroring the existing `_safety_gate` pattern.

### [P1] — 8 inline raw-SQL sites bypass the L2/L13 storage lanes — engine.py:255, 269, 301, 335, 358, 416, 1096, 1103 — why: the composition root writes/reads `engine_state` (pause), `orders` (status updates, idea_id link), and `ideas` directly via `self.conn.execute`. INTERFACES §10 says lanes contribute helpers (`insert_row`/`supersede_row`/idea_store/position_store) and the engine composes them; here the engine has absorbed persistence logic that belongs in `db/` or a new `engine_state_store`. The pause INSERT...ON CONFLICT and the orders UPDATE are real schema knowledge living in the orchestrator. — recommended refactor: move `engine_state` read/write into a tiny `db/engine_state.py` (`load_pause`/`save_pause`); move order-status transitions into a `order_store.mark_status(conn, order_id, status)`; move the idea_id link into `idea_store`/`order_store`. Engine keeps the *decision*, not the SQL.

### [P1] — `_reconcile_pending_orders` (118 lines) mixes query, terminal-mapping, buy-advance, sell-close-out, logging, and audit — engine.py:283-400 — why: it is a second mini-orchestrator inside the orchestrator. It loops rows and branches into `_mark_order_terminal`, `_close_out_filled_sell`, and an *inline* buy→MONITORED idea advance (the very (ticker,bucket) SQL of the P1 above). The reconciler is conceptually a sub-project (§10 `023_orders_idea_id`, sub-project #2) that has no module of its own — it lives as engine methods. — recommended refactor: extract an `OrderReconciler` (in `execution/` alongside `submit.py`/`exit_monitor.py`) taking `(conn, executor, pit, clock, advisor_id_for, audit_path, metrics)`. Engine calls `reconciler.run(now)`. This also relocates `_mark_order_terminal`, `_close_out_filled_sell`, `_sell_label_kind` out of the Engine dataclass.

### [P2] — `_build_learning_inputs` (96 lines) embeds the entire trust/calibration backtest-vs-live policy in the engine — engine.py:687-782 — why: the engine knows `TrustLedger.should_update`, `MultiAdvisorCalibrator` construction, per-advisor `Calibrator(...).fit/persist`, the backtest "recompute every step / no cache" rule (D2), the live cache tuple shape, and the warm-start fallback (D7). This is sub-project #4's policy living in the composition root; `_learning_cache` + `calibrators` + `ledger` are stateful engine members purely to serve it. — recommended refactor: extract a `LearningStep`/`TrustCalibrationCoordinator` (trust lane or a new `learning/` module) owning `ledger`, `calibrators`, `_learning_cache` and exposing `build_inputs(conn, now, advisor_map, backtest) -> (WeightBundle, calibrator)`. Engine holds one injected coordinator instead of three learning fields + 96 lines.

### [P2] — `build_engine` (143 lines) is a long linear wiring script with branching policy embedded — engine.py:1261-1403 — why: it is mostly fine (this is where DI *should* live) but it has grown branching policy: the current-price/market-calendar gate (backtest vs alpaca_paper, amendment C0), the executor type assertion + sim-only seed, the audit-clock monkeypatch (`_audit_mod._clock = lambda ...`, line 1317). The C0 gate in particular is real business logic (PIT-purity boundary) inlined in the factory. — recommended refactor: extract `_build_live_data_providers(config, clock) -> (CurrentPriceProvider, MarketCalendar)` encapsulating the C0 gate, and `_build_executor_seeded(config, conn)` for the executor+seed. `build_engine` becomes a flat list of `x = build_x(...)`.

### [P2] — Market-hours heuristic `_us_market_open` lives as a module function in the orchestrator — engine.py:108-123 — why: a coarse UTC-offset market-hours guess (no DST/holidays) sits in engine.py while a real `MarketCalendar` (AlpacaMarketCalendar/OfflineMarketCalendar) already exists in `runtime/market_calendar`. `_market_is_open` already prefers the injected calendar; the heuristic is now a legacy fallback misplaced in the engine. — recommended refactor: move `_us_market_open` into `runtime/market_calendar.py` as the `OfflineMarketCalendar` fallback impl; engine calls only `self.market_calendar`. Removes ~16 lines + a `datetime`-math concern from the orchestrator.

### [P2] — `_build_a1_insider_fn` / `_build_a1_congress_fn` are near-identical and encode advisor knowledge in the engine — engine.py:130-187 — why: the two builders differ only in `s.source == "form4"` vs `"congress"` and the advisor_id. detect→filter→max-by-conviction→score→emit is advisor wiring that belongs with the signals/advisor lane, not the composition root. Adding A2/A3 means more copies here. — recommended refactor: collapse to one `_build_source_advisor_fn(advisor_id, source, db_path, clock)`; better, move advisor construction into a `signals`/advisor registry the composition root just imports. (`pit` is accepted but unused in both — dead param.)

### [P3] — Test-only attribute set via `# type: ignore[attr-defined]` on CycleResult — engine.py:614, 627, 639, 1156 — why: `_r.paused_by_alert = True` monkeypatches an attribute onto `CycleResult` that the dataclass does not declare (4 sites). This is a contract smell: the pause signal is bolted on rather than a field. — recommended refactor: add `paused_by_alert: bool = False` to `CycleResult` (orchestrator lane) so the type-ignores disappear and the field is discoverable.

### [P3] — Broker-fatal signalling via mutable-list closure side-channel — engine.py:1048, 1082, 1155 — why: `_broker_fatal: list[str] = []` is a closure-captured mutable used to smuggle a BrokerError signal out of `_bound_submit` back to `run_cycle` scope because `cycle.py` swallows the exception. Works, but it is an implicit control-flow channel that the `run_cycle` split (P0) should formalize. — recommended refactor: have the extracted submit step return a small `SubmitOutcome(filled, broker_fatal)` rather than mutating a captured list.

## OPPORTUNITIES TO ADD

- **Split engine.py along its seams.** The module already *names* its sub-projects in docstrings (#2 exit monitor, #3 daemon/market-hours, #4 learning, #5a opinion persistence). Make those module boundaries real: `execution/order_reconciler.py`, `learning/coordinator.py`, `db/engine_state.py`, `signals/advisor_factory.py`. Target: `engine.py` ≤ ~400 lines holding only `Engine` (thin conductor) + `build_engine`.
- **A single `advisor_id_for` / attribution source of truth** shared by engine + daemon (kills the ×4 duplication and the future A2/A3 mis-attribution risk in one move). This is the highest correctness leverage per line changed.
- **`idea_store.resolve_idea_for(...)`** as the one sanctioned (ticker,bucket)→idea resolver; ban inline ideas-table SQL in the engine (a CI grep like the §11 convention greps could enforce "no `FROM ideas` outside `idea_store`").
- **Promote `paused_by_alert` to a real `CycleResult` field** and delete the 4 `type: ignore` monkeypatches.
- **Characterization tests before refactor:** `run_cycle` has 9 closures and order-dependent steps (R6 sweep-last). Add a golden/characterization test of the full cycle (ideas in → orders/outcomes/audit out) FIRST so the extractions are provably behavior-preserving.
- **Co-locate the reconcile+fail-closed preamble** with `_safety_gate` as the shared `_preflight*` the two daemon entrypoints both call, closing the last drift gap between `run_cycle` and `run_fast_iteration`.

---

*DI verdict:* The "lanes never import each other; everything composed in `build_engine`" contract is **still honored at the import graph level** — no lane→lane imports, all cross-lane wiring is constructor-injected or passed as callables. The erosion is internal to the composition root: it has absorbed business logic (advisor attribution, the ideas join, learning policy, engine_state persistence, market-hours) that belongs in the lanes it composes. Refactor target is *extraction out of engine.py into existing lanes*, not re-architecting the DI.
