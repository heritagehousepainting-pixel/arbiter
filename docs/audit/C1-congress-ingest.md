# C1 — Congress Ingestion Correctness Audit

**Lane:** C1 (congress ingestion correctness) — READ-ONLY audit
**Scope:** `arbiter/arbiter/ingest/congress/` — House `{year}FD.zip` index → PTR PDFs (pdfplumber); Senate efdsearch CSRF/cookie/DataTables → HTML tables. Idempotency/amendments = C3; fallbacks = C4 (referenced but not owned here).
**Date:** 2026-06-19
**Auditor:** automated code+test review, no live network.

## Verdict

**PASS WITH RESERVATIONS.** The two contract invariants most central to signal correctness are upheld: amounts are preserved as `(amount_low, amount_high)` and are **never midpoint-imputed** (`normalize.to_raw_filings` copies `txn.amount_low/high` verbatim; `parser._AMOUNT_BRACKETS` stores raw bracket bounds), and `filing_ts` is derived from the **notification/disclosure date**, not the transaction date (`_notification_date_to_filing_ts(notification_date)`, asserted by `test_congress_normalize.py:113-141`). PTR filtering (`FilingType=="P"`), non-equity/CUSIP filtering, and ticker resolution are implemented and tested. However, several parse paths can **silently drop or misattribute** transactions, and one `filing_ts`-semantics gap (House uses the per-row Notification Date rather than the index FilingDate) can shift the information timestamp. No P0. Findings below are P1–P3.

---

## Findings

### P1 — House `filing_ts` uses per-row Notification Date, ignoring the actual index FilingDate — `normalize.py:155-160` + `__init__.py:91-133` — `IndexRecord.filing_date` (the real disclosure/clerk-receipt date) is parsed in `index.py` but is **never propagated** into the `Transaction`; `filing_ts` is built solely from the PTR PDF's per-row "Notification Date." The Notification Date is when the member says they were *notified* of the trade, which can be **earlier** than the date the filing was actually received/published by the Clerk (`FilingDate`). The information only becomes public (and thus actionable, no-look-ahead) at `FilingDate`. Using the earlier notification date overstates how early the signal was knowable and can introduce look-ahead in PIT joins. — **Recommended action:** thread `IndexRecord.filing_date` through to `to_raw_filings` and set `filing_ts = max(notification_date, index_filing_date)` (or the index FilingDate outright), keeping notification_date in `raw_json` for provenance.

### P1 — Senate ticker falls back to free-text cell content, admitting non-tickers — `senate.py:381-396` — `_parse_ticker` returns `cell_text.strip()` when no `<a>` tag is present. If the Ticker column ever contains a stray label, a company fragment, or punctuation (anything that is not `--`/empty), it is passed downstream as a ticker. `normalize._VALID_TICKER_RE` is **defined but never applied** (only `_is_cusip_shaped` runs), so a malformed token like `"N/A"`, `"None"`, or `"ACN -"` survives normalization and becomes a fabricated tradeable signal / misattributed ticker. (House's `parser.py:194` guards `{"N/A","NONE","--"}`; the PTR-PDF + Senate path does not.) — **Recommended action:** in `normalize.to_raw_filings`, reject tickers that fail `_VALID_TICKER_RE` (after CUSIP check); and in `_parse_ticker`, only accept the bare-text fallback if it matches a ticker shape, else return `None`.

### P1 — `_parse_txn_type` defaults unknown Senate types to "S" (sale) — `senate.py:409-422` — Any transaction-type cell that is not an exact known string ("Purchase"/"Sale (Full)"/"Sale"/"Sale (Partial)"/"Exchange") is silently coerced to `"S"` and kept. A new label, a localization change, or an "Exchange"-variant string would be **misattributed as a sale** rather than dropped. This is a directional-signal corruption (a buy or an ambiguous row entering the engine as a sell). — **Recommended action:** map unknown types to a sentinel (e.g. `"?"`) that `to_raw_filings` drops via the `_KEEP_TXN_TYPES` filter, instead of defaulting to `"S"`.

### P2 — House amount extraction can grab the wrong two dollar figures — `ptr_pdf.py:297-315` — `_extract_amounts` first tries the contiguous `$lo - $hi` pattern, then **falls back to "first two `$` amounts found anywhere"** in `post_date` (and finally the whole body). On rows where the amount range wrapped and an unrelated dollar figure (e.g. a `$200` cap-gains threshold leaking through, or a `$50,000` continuation token) appears before the true high bound, the fallback can pair the wrong numbers, silently producing a wrong `(low,high)`. The `$200?` skip pattern (`_SKIP_PATTERNS`) mitigates but is line-anchored and brittle. — **Recommended action:** prefer the contiguous pattern only; if it fails, log and skip rather than guessing first-two-dollars, or constrain the fallback to dollar tokens after the dash.

### P2 — Senate column mapping is positional with no header validation — `senate.py:506-536` — `_parse_ptr_page` hard-codes `cells[1]=txn_date, [3]=ticker, [4]=asset_name, [5]=asset_type, [6]=txn_type, [7]=amount`, guarded only by `len(cells) < 9`. If the eFD table adds/reorders a column (it has historically changed), every field **silently shifts** — e.g. asset_type read from the wrong column, amounts from the comment column — and rows still parse (no exception), so transactions are misattributed rather than dropped. — **Recommended action:** parse the `<thead>` and map columns by header label, or assert the header row matches the expected sequence and skip the whole report (logged) on mismatch.

### P2 — House type-token detection can misfire on asset names containing standalone P/S/E — `ptr_pdf.py:352-364` — When no `S (partial)` is present, the parser scans `pre_date` for the **last** standalone `P`/`S`/`E` token (via lookbehind/lookahead). An asset name ending in a lone capital that the regex treats as standalone (e.g. a trailing share-class letter or an abbreviation followed by a non-excluded delimiter) could be misread as the transaction type, splitting `asset_raw` wrongly and yielding a wrong `txn_type` / truncated asset name. Real fixtures pass, but the heuristic is fragile against unusual asset names. — **Recommended action:** anchor the type token to its expected position (immediately preceding the date pair) rather than "last standalone P/S/E anywhere in pre_date."

### P3 — Senate `is_amendment` detected only via the word "Amendment" in link HTML — `senate.py:315` — `bool(re.search(r"Amendment", link_html, re.IGNORECASE))`. If the eFD label changes ("Amended", "(A)", a separate column flag), amendments are recorded as `is_amendment=False`, which C3's supersede logic depends on. (Flagged here as a correctness input; supersede mechanics are C3.) — **Recommended action:** corroborate with the report-type code from the DataTables payload, not just the rendered link text.

### P3 — House CUSIP guard's second rule may over-drop short numeric-led tickers — `normalize.py:172-183` — Rule 2 drops any ticker that starts with a digit and is `len > 5`. Legitimate symbols are rarely numeric-led, so risk is low, but combined with the CUSIP rule this is the only ticker-shape gate and there is no positive `_VALID_TICKER_RE` allow-list applied (see P1 finding above). Documented for completeness. — **Recommended action:** fold into the P1 fix by switching to an explicit allow-list + CUSIP deny-list.

### P3 — `parse_index` accepts ≥9 fields but silently ignores extras; tab-in-field would shift columns — `index.py:113-134` — Rows are `split("\t")` and unpacked positionally with `*_extra`. A literal tab embedded in a name/state field would shift `FilingType`/`DocID` and the row would still parse (wrong `filing_type`, possibly dropped from PTR filter or misclassified). Low likelihood for the Clerk's machine-generated file. — **Recommended action:** none required; optionally assert exactly 9 fields and log when `_extra` is non-empty.

## OPPORTUNITIES TO ADD

- **Apply `_VALID_TICKER_RE` as a positive gate.** It already exists in `normalize.py:139` but is dead code. Wiring it in closes both the Senate free-text-ticker hole (P1) and the House numeric-led edge (P3) in one place.
- **Propagate index FilingDate into the House Transaction** so `filing_ts` reflects public availability and `raw_json` retains both dates (addresses P1, improves PIT auditability).
- **Header-driven Senate column mapping** (parse `<thead>`) — makes the parser self-validating against eFD layout drift instead of silently misattributing.
- **Drop-counter / reconciliation metric.** Emit counts of (rows seen, rows dropped per reason) per doc_id from `to_raw_filings` and the senate/PDF parsers, so silent drops become observable rather than buried in DEBUG logs.
- **Amount-bracket cross-check.** `parser._AMOUNT_BRACKETS` enumerates the canonical STOCK Act brackets; `to_raw_filings` could optionally snap/validate parsed `(low,high)` against this fixed set and flag any pair that is not a known bracket (catches P2-style mis-pairing).
- **"Over $50,000,000" high-bound is a sentinel equal to low** (`parser.py:68`) — document/standardize how downstream treats the open-ended top bracket so conviction sizing does not treat `high==low` as a degenerate zero-width range.
